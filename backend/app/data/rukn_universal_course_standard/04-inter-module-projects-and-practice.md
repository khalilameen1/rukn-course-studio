# 04 INTER MODULE PROJECTS AND MODULE CLOSURES

## Product boundary

This file designs the short learner action spoken by the lecturer between modules
It stops at the learner task and upload form

The generated product contains

- One compact project after every non final module
- The last project before the final module designated internally as the compact integrated readiness proof
- The exact image screenshot or file the learner should upload
- One natural unlabelled closure immediately after that project

The generated product contains no platform or post upload operations
Keep all authoring checks internal

## Placement contract

Every non final module ends in this order

1. The last teaching reel completes its promise
2. One compact project requires the learner to apply the dominant module exit capability
3. One to three natural spoken sentences close the achieved capability and expose the next real performance condition
4. The following module opens by resolving that same condition

Do not place a formal upload project in the middle of a module
Lessons may still contain immediate no upload practice

Do not add a project after the final module
The final module ends with a complete course resolution and no forward bridge
The project at the last genuine boundary before the final module is the compact integrated readiness proof
The final module uses that work to diagnose compare refine transfer package communicate deliver hand off or close the performance without introducing a new promise critical standalone capability

## Final inter module readiness project

For a course with at least two modules the final formal upload is the project immediately before the final module
Assign it the internal role `FINAL_INTEGRATED_READINESS`
Do not print that role in the lecturer DOCX unless an ordinary learner friendly project title naturally fits the content

This project must

- Use a bounded realistic situation input conversation case task teaching moment or artefact
- Reuse prior learner work or a course supplied starting point whenever that prevents irrelevant rebuilding
- Require synthesis and independent judgement rather than a longer repetition of an earlier exercise
- Represent most core capabilities
- Represent every promise critical capability whose absence would make the readiness claim false
- Stay inside one to three uploads and the compact effort policy whenever possible
- Produce one directly inspectable integrated result
- Be fully solvable from capabilities taught and practised before the boundary

This project must not

- Become a multi hour portfolio capstone
- Require all minor facts to be repeated when the integrated action already depends on them
- Introduce a capability reserved for the final module
- Depend on a real client live account delayed metric public response or external approval

The final module may diagnose refine transfer package communicate deliver hand off or close this work using already introduced capability loops
When the final module contains the first teaching of a promise critical action move that action and its practice before this project or revise the promise
Do not leave the course with an unproved final capability merely to preserve module order

## Compactness and effort contract

Use the smallest authentic task that makes the most important completed capability observable in its native performance form
At the final boundary use the smallest bounded integrated episode that makes the accepted promise critical capability set observable together
The project may come from any domain
Choose its action and attachment from the capability adapter rather than from the course title

Default module project constraints

- Ten to twenty five minutes of expected active learner effort
- Mandatory redesign review above thirty minutes
- One clear situation
- One dominant action or the smallest inseparable action cluster for a standard boundary
- One connected integrated performance episode rather than unrelated task accumulation at the final boundary
- One to three uploadable outputs
- One primary output whenever possible
- Inputs supplied by the course when searching for inputs would test an unrelated skill
- Tools actions and decisions already taught
- No multi day waiting future metric client reply or external approval
- No long report when one image screenshot or compact file exposes the same work

A checkpoint is not a portfolio case
It preserves the real decision action constraint and result without reproducing the full size of paid work

When the completed capability is too broad

1. Identify the smallest performance later work genuinely depends on
2. Make that performance the current project
3. Move other capability use to later projects
4. Reuse the same learner work across boundaries when that reflects real workflow

Do not compress until the task becomes decorative
Do not expand until the learner is doing unrelated busywork

## Upload form

The platform categories are

- `IMAGE`
- `SCREENSHOT`
- `FILE`

`FILE` may contain a PDF short document spreadsheet presentation editable source code configuration data archive audio video log test result or another compact artefact native to the capability

Name the required form in the lecturer speech
Do not say only upload your work
State what the learner should attach and what must be visible in it

Use the fewest attachments that can honestly show the action
Do not ask for several screenshots when one complete readable view is enough
Request an editable or structured file only when structure formulas layers timing code organisation or editability are part of the capability

The attachment must be understandable as evidence of the learner action without

- An external link
- A private account
- A live dashboard
- A real client response
- Hidden context
- A result that appears only after waiting

A static image cannot represent pronunciation timing interaction movement runtime behaviour or another capability that exists only in performance
Use an appropriate file when the platform supports it
When the available forms cannot show the whole capability narrow the checkpoint to an honest observable precursor rather than pretending the full performance is visible

## Attachment self evidence check

Before writing the spoken project block run `ATTACHMENT_SELF_EVIDENCE_CHECK` on the planned submission
Its only purpose is to verify that the submitted image screenshot or file itself contains the decisive result needed to understand the learner action
This is an authoring inspectability check rather than an evaluation design

```yaml
attachment_self_evidence:
  check_id:
  project_id:
  supported_file_form:
  file_can_be_opened_and_inspected:
  decisive_result:
  decisive_result_location:
  required_visible_audible_readable_structural_or_recorded_runtime_contents: []
  attachment_form_matches_the_capability: true
  external_link_required: false
  private_account_or_live_dashboard_required: false
  hidden_context_required: false
  unseen_process_inference_required: false
  long_explanation_required: false
  ambiguity_risk:
  ambiguity_repair:
  rubric_or_evaluator_logic_generated: false
  status: PASS | REDESIGN_PROJECT
```

`decisive_result_location` may identify a page sheet slide filename frame region time range code file test output or another compact location inside the submitted artefact
Use it internally to prevent vague or unreadable submissions
Do not export this locator as a scoring criterion
Only tell the learner what must be present when that instruction is needed to create the correct file

Pass only when

- The attachment can be opened in a platform supported form
- The learner action is represented directly rather than guessed from intent
- The decisive result is visible audible readable structurally inspectable or represented by a compact recorded runtime or test output appropriate to the capability
- The file is understandable without an external link private state delayed result or missing brief
- Any required context is contained in the project instruction supplied input or attachment itself
- The evidence form represents the real capability rather than an easy proxy

This check must never generate

- A rubric
- Scores thresholds pass fail rules or grading weights
- An evaluator prompt or chain of instructions
- Feedback revision routing or retry policy
- Safety moderation privacy or LMS operations
- Any text that appears after the learner upload instruction in the lecturer DOCX

When the check fails redesign the task attachment form or scope
Do not compensate with a long learner explanation of invisible work

## Capability native project evidence

Use the accepted active adapter and context overlays to decide what the learner must upload

- Knowledge may use retrieval explanation classification or bounded case application
- Perceptual discrimination must include the learner judgement and the cue that controlled it
- Language speaking listening live interaction teaching or physical action normally requires an audio or video file
- Operational work may use a completed case record state screenshot or structured file
- Creation may use an image export or editable source when structure matters
- Technical work may use source logs tests configuration or runtime evidence
- Analysis may use a spreadsheet notebook or short decision record with visible inputs and reasoning

When a standard boundary capability is composite choose one dominant performance and include only inseparable secondary proof
At the final boundary choose one connected integrated performance episode and include every promise critical mode without adding unrelated outputs
A call centre language checkpoint may use one short audio or video response plus one compact case note file when both spoken handling and documentation were taught and later work depends on both
A sales checkpoint may use one role play recording when discovery response adaptation and close are the completed cluster
A teaching checkpoint may use one short teaching simulation when modelling elicitation correction and transfer are inseparable

Do not require every mode used anywhere in the course to appear in every project
Do not force all projects into screenshots because the platform labels audio and video under `FILE`
Do not create grading rubrics evaluator instructions scores or post upload logic in the project text

## Project progression

Across the course increase one or more of

- Independence
- Input realism
- Competing constraints
- Ambiguity
- Failure diagnosis
- Quality responsibility
- Adaptation
- Learner customer client stakeholder counterpart system or handoff judgement

Do not repeat the same action at the same level under a new title
Merge or redesign duplicate checkpoints

A standard project checks the module that has just ended
The final boundary project checks the accumulated promise critical capability set including the module that has just ended
No project may require the learner to solve an untaught capability from the following module
Later teaching may depend on the completed output or expose where it stops being sufficient

The last boundary project is different in breadth but not in workload philosophy
It integrates previously practised capabilities through reuse and a bounded situation rather than adding unrelated volume
Its `promise_critical_capabilities_omitted` list must be empty before the final module is accepted

## Authenticity without unnecessary difficulty

Use real performance friction only when it teaches judgement

Useful compact constraints include

- One missing ambiguous noisy or imperfect input that changes the decision or response
- A fixed format time resource or delivery condition
- Existing material that must be used
- Two competing priorities
- A common learner customer client stakeholder counterpart or system signal
- A weak result that needs one diagnosis and correction
- An output that must be adapted instead of rebuilt
- A handoff condition the learner must satisfy

Do not create a long fictional client biography
Do not add arbitrary obstacles to make the project feel serious
Do not require paid spend public posting recruitment or access to a real client

Domain safety legal or quality constraints belong in the project when they are part of the taught capability
Platform and post upload procedures do not belong in the course script

## Inputs access and cost

The project must state where its inputs come from
Prefer supplied synthetic public domain or clearly available inputs so the checkpoint measures the module capability rather than luck or search ability

For a paid tool dependency

- Require it only when it is essential to the promised professional output
- Use a durable available alternative for learning when it teaches the same principle honestly
- Do not claim that a limited substitute produces an identical professional result
- Do not require simultaneous subscriptions that the task does not need

Never place exact subscription prices client service prices current offers temporary plan limits or similar fast expiring details in the spoken project
State the required function and a durable access condition instead

## Spoken project block

Write the project as natural lecturer speech addressed only to the learner

Include only

- The situation or starting point
- The completed capability being used
- The supplied input stimulus utterance case or material
- The exact perception response explanation decision or action
- One or more meaningful constraints
- The image screenshot or file to upload
- What must be visible or audible in that upload

Do not include

- Internal project field names
- Research or source notes
- Platform or post upload operations
- A recap of the whole module
- An unavailable companion brief checklist workbook or template
- A promise that another file will be supplied later
- Fast expiring prices rates plan details rankings interface paths temporary features or current statistics

The spoken block must be self contained inside the lecturer DOCX
A pre existing platform asset may be named only when it actually exists before generation

## Module closure construction

Place the closure as the last spoken lines of the project block
Do not give it a heading
Do not label it bridge transition cliffhanger or conclusion

A valid closure performs three functions without sounding formulaic

1. Confirms what the learner can now do or what the completed output establishes
2. Introduces one honest condition limit consequence or pressure that appears in the real performance
3. Stops after making that later capability necessary without announcing the syllabus

The source of forward pull may be

- The result works until the input changes
- The output is correct but still needs inspection under a harder condition
- One decision solves the simple case but creates a tradeoff at scale
- A learner customer client stakeholder counterpart or system response changes what counts as correct
- Separate parts work but have not yet been integrated
- The learner can execute but cannot yet diagnose a deceptive failure
- The work is complete internally but not ready for professional delivery

Do not use

- In the next module
- In the following part
- We will learn later
- Wait until you see
- The surprise is
- Are you ready
- But that is not all
- Artificial danger mystery urgency or exaggerated stakes
- A question used only as a repeated teaser formula
- Withholding any answer promised by the completed module

The bridge is earned only when the newly exposed condition is real and the following module actually resolves it
A quiet technical limit is stronger than invented suspense

## Closure variation

Do not repeat one verbal shell at every boundary
Vary the natural function according to the work

Possible shapes include

- Result then limitation
- Completed decision then changed condition
- Successful output then inspection pressure
- Independent action then integration need
- Correction then transfer challenge
- Completion readiness then counterpart stakeholder delivery or transfer consequence

Track repeated openings connectors and sentence shapes in `PHRASE_LEDGER`
Reject a course in which every module ends with the equivalent of now you can do this but there is still one problem

The final module receives no bridge
Its ending resolves the promise reflects the learner capability honestly and stops without a teaser

## Spoken length

Project instructions must be complete without becoming another module

Calibration ranges

- Standard project block including closure 100 to 260 spoken words
- More integrated final boundary project including closure 180 to 380 spoken words
- Absolute spoken ceiling five minutes

These are not padding targets
Use the number of words the action actually needs

## Project and closure gates

Block the map or script when

- A non final module has no project
- A project exists after the final module
- The project can be completed without the dominant module capability
- A standard project tests many loosely connected lessons instead of one exit capability or the smallest inseparable cluster
- A final boundary project combines unrelated outputs instead of one connected integrated performance episode
- Inputs tools actions constraints upload form or visible output are unclear
- Expected effort exceeds thirty minutes without a capability based reason and redesign
- It requires a real client external account approval paid spend public response lucky input or delayed result
- It requires more than three uploads without proof that fewer cannot show the capability
- The uploaded result does not visibly audibly readably structurally or through an appropriate compact recorded runtime result represent the learner action
- The internal attachment self evidence check is missing fails requires hidden context external access unavailable runtime execution or has been turned into evaluator grading or post upload logic
- A static output is used for a live auditory interactive teaching physical or adaptive capability it cannot represent
- A standard project attachment type comes from the course family rather than the dominant capability adapter
- A final boundary attachment set fails to represent one or more promise critical adapters in the connected integrated episode
- The project uses a provisional rejected or unresolved custom adapter
- A required safety privacy authority tool stakeholder language or access overlay is absent from the task boundary or evidence
- A composite project omits an inseparable secondary mode required by later work or includes unrelated modes that inflate effort
- A supplied example can be copied unchanged and still complete the task
- The module did not teach how to produce or inspect the requested result
- The project duplicates another checkpoint at the same level
- The last project before the final module is not assigned `FINAL_INTEGRATED_READINESS`
- The final integrated readiness project omits a promise critical capability or becomes an oversized capstone
- The final module introduces a new promise critical standalone capability whose honest proof would require another upload
- The final module does not use the final integrated readiness work or exists only as recap
- The following module depends on a capability this project never uses
- The project requires knowledge from the following module
- The spoken block reteaches the module omits the upload form or exceeds five minutes
- The spoken block contains platform or post upload operations
- The spoken block contains a fast expiring detail
- The closure is missing labelled theatrical formulaic or explicit about what comes next
- The closure opens a problem the following module does not resolve
- The closure withholds the current module answer
- The final module ends with forward pull instead of complete resolution
