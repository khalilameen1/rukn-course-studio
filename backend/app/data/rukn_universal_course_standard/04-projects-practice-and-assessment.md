# 04 PROJECTS PRACTICE AND ASSESSMENT

## Project spine rule

Projects are the evidence path of the course
Design them before finalising the lesson map

Each module ends with one project gate unless the product explicitly uses another gate policy
In the Rukn teleprompter export project instructions appear only between modules
Do not interrupt the spoken lesson sequence with decorative mini assignments

Lessons may contain immediate micro practice but formal submissions belong to project gates

Project gates are the real act boundaries of the Rukn progression
They convert accumulated explanation into proof expose the current limit and earn the next module
Do not label or perform them as dramatic acts in the learner facing text
The learner should feel a genuine change in capability and consequence rather than a decorative pause between content blocks

## Project validity

A valid project produces an observable learner performance or deliverable under meaningful constraints

It must include

```yaml
project:
  real_world_role:
  situation:
  inputs:
  constraints:
  required_decisions: []
  required_actions: []
  deliverables: []
  quality_evidence: []
  failure_checks: []
  allowed_tools:
  submission_format:
  assessment_rubric:
  capabilities_reused: []
  new_independence:
```

Reject projects that ask the learner only to summarise list explain choose a favourite or imitate a finished example

## Practice ladder

Move through these degrees as appropriate

1. Observe and distinguish
2. Reproduce with guidance
3. Complete with constraints
4. Diagnose a weak result
5. Revise using evidence
6. Adapt to a changed context
7. Make an independent tradeoff
8. Deliver under professional conditions

Do not jump from explanation to an unconstrained capstone
Do not keep the learner in imitation for the whole course

## Module project progression

Each project must add at least one of

- Greater independence
- More realistic inputs
- More competing constraints
- More ambiguous evidence
- A new failure mode
- A tighter quality bar
- A professional communication or handoff requirement
- Transfer to a different context

If two projects test the same thing at the same level merge or redesign them

After each project record

- What capability is now proven
- What failed or remained limited under the new conditions
- Why the next module is now necessary

Do not begin the next module with an unrelated topic merely because it belongs to the syllabus

## Authenticity without artificial difficulty

Use realistic messiness only when it teaches judgement
Do not add irrelevant work to make a project feel serious

Good constraints resemble actual work

- Incomplete or imperfect brief
- Fixed budget or time
- Existing assets
- Conflicting priorities
- Platform or device limitations
- Client feedback
- Quality or compliance requirements
- A result that must be adapted rather than rebuilt

Bad constraints are arbitrary obstacles unrelated to competence

## Accessibility and cost

The core project must be possible with the minimum access defined in the thesis
If a professional paid tool is important

- Provide a learning path using a trial free tier open alternative sample data or supervised demonstration when possible
- State internally what cannot be replicated without the professional tool
- Do not pretend an inferior substitute produces identical professional output
- Do not require simultaneous subscriptions that are not needed for the project

## Inputs and assets

Projects must specify where inputs come from

- User supplied
- Course supplied
- Publicly licensed
- Learner collected with consent
- Synthetic and clearly identified

Do not instruct learners to use unknown assets confidential data or material without suitable rights
Do not make project completion depend on finding lucky inputs

## Rubric design

Use observable criteria tied to the capability graph
Avoid vague criteria such as creative professional clean good or engaging without evidence

Each criterion should contain

```yaml
criterion:
  observable_evidence:
  pass_condition:
  common_failure:
  diagnostic_question:
  correction_direction:
```

Weight critical functional decisions above decorative polish
Separate factual correctness process quality output quality and professional delivery where relevant

## Feedback contract

Feedback must identify

1. What evidence was observed
2. Which decision or execution caused it
3. Why it matters to the promised task
4. The smallest correction with the highest effect
5. What the learner should test after correction

Do not give taste based feedback as universal truth
Do not rewrite the whole project for the learner
Do not praise effort instead of assessing evidence

## Revision loop

Important projects require at least one inspect and revise cycle
Professional competence is not proven by a lucky first attempt

Store

- First version
- Feedback or test result
- Decision log
- Revised version
- Short explanation of the changed decision

For language performance store first attempt targeted correction and second performance
For code store failing behaviour diagnosis change and passing verification
For operational skills store initial decision outcome signal diagnosis and next decision

## Portfolio evidence

For career oriented courses select projects that can become credible proof of work

A portfolio case should reveal

- The problem and context
- The learner role
- Constraints
- Key decisions
- Process evidence
- Result or deliverable
- Revision based on evidence
- What is real simulated or unverified

Do not invent client names performance numbers revenue reach conversion or testimonials
For simulated work state that it is a training case

## Capstone contract

The capstone must test the course promise as a connected performance
It must require the learner to retrieve and combine earlier capabilities without step by step imitation

The capstone must include

- A realistic brief or situation
- More than one plausible path
- Explicit constraints
- Required decision records
- Execution
- Quality inspection
- Revision
- Professional delivery
- A rubric mapped to the thesis

Do not make the capstone merely bigger
Make it more integrated independent and accountable

## Project instruction voice

Project instructions in the teleprompter are spoken by the lecturer
They must sound direct practical and specific

Include enough detail for independent execution
Do not hide required steps in dense formal prose
Do not pad with encouragement
Do not repeat all preceding lessons
Refer to the capability being proved and the evidence required

## Spoken project length

Project instructions must be complete without becoming a second module

Useful spoken defaults

- Standard module project 250 to 700 words
- Complex integrated module project 500 to 900 words
- Final capstone 700 to 1200 words

These are calibration ranges not padding targets

When the project needs long asset specifications rubrics datasets reference tables submission checklists or technical requirements place those in a separate learner project brief and keep the teleprompter explanation focused on the situation decisions workflow evidence and submission

Do not repeat lesson teaching to increase project length
Do not compress a project until essential constraints or pass evidence disappear

Outside these ranges require an explicit reason and a speakability review

## Project gates

Block a module when

- The project can pass without the module capability
- Required inputs or tools are unavailable to the defined learner
- Success depends on subjective taste without a rubric
- The project asks for a result but teaches no inspection method
- The project invents business results
- The submission cannot be assessed from the stated evidence
- The project duplicates another gate
- The next module depends on competence that this project does not verify
- The next module is not earned by a capability limit consequence or transfer need exposed by this gate
- Spoken instructions reteach the module or exceed the useful project explanation without justification

Block course completion when the capstone does not prove the observable end performance
