# 09 SINGLE DOCX DELIVERY CONTRACT

## Sole user visible course deliverable

Export exactly one course file

`FINAL_LECTURER_TELEPROMPTER.docx`

The DOCX contains only

- Course title
- Optional short course subtitle
- Module headings
- Reel headings
- Lecturer spoken text
- Module project headings and spoken project instructions between modules
- The natural unlabelled module closure bridge as the final spoken lines of each non final project section

Do not export a course map research or source pack project companion file production guide reviewer report or any second document
Do not return internal planning artefacts as separate files
The map exists to produce the script
It is not a second product

## Learner facing content boundary

The final DOCX may tell the learner

- What to do
- What input to use
- What constraint matters
- What result to produce
- Whether to upload an image screenshot or file
- Which exact item to upload
- What must be present so the learner knows the task is complete

The lecturer speaks only to the learner
Each project ends when the learner knows exactly what to make and upload
Anything that happens after upload belongs outside this standard and outside the teleprompter
Internal attachment self evidence records final boundary coverage records and final module resolution records are authoring controls only and never appear in the DOCX

## Spoken continuity

Within each module the lecturer text flows reel by reel without production commentary
A project section closes the module and separates it from the following module

The final lines of every non final project section complete the current capability then expose one real next work condition
They are spoken naturally and carry no label such as hook loop cliffhanger bridge or transition

Do not insert

- Learning objectives as administrative bullets
- Lesson summaries that the lecturer will not say
- Recap pages
- Source pages
- Visual plans
- Reviewer reports
- Estimated timing
- Hook loop tension payoff escalation or episodic progression labels
- Camera notes
- Screen notes
- Alternative takes
- Platform operations unrelated to the taught skill
- Update notices or recording date disclaimers

If the lecturer must say an instruction write it as natural speech
If production or platform operation needs an instruction keep it outside this course generation standard

## Single document self containment

The generated course must be usable for recording from the DOCX alone

Do not require a separately generated

- Learner workbook
- Project brief
- Checklist
- Source appendix
- Glossary
- Asset guide
- Platform operations guide
- Course map

All essential project actions constraints output descriptions and upload instructions must appear in the spoken project section

A pre existing external asset may be referenced only when it was supplied before generation or already exists in the platform
Name it exactly in the spoken instruction
Do not invent an asset and write as though it will be delivered later

Internal demonstration plans and source records may guide generation
They are not exported and may not be required by the lecturer to understand what to say

## Strict evergreen delivery boundary

The final DOCX must not contain fast expiring content

Reject or rewrite

- Exact service prices
- Exact client fees presented as current market truth
- Subscription prices
- Plan names tied to current prices quotas discounts or temporary features
- Current salaries rates exchange rates advertising costs or platform fees
- Current rankings best tool lists trend claims or temporary algorithm advice
- Current follower counts statistics or adoption numbers used as teaching proof
- Temporary interface positions fragile menu paths or release specific steps
- Current limits thresholds or policy values likely to change
- Date bound statements that make the recording stale

When a changing value matters teach the stable variables decision method consequence and live verification habit
Do not solve expiry by adding a disclaimer that the number may change
Do not place the volatile value in another delivered file

## DOCX formatting contract

Use a simple stable hierarchy

- Right to left document direction for Arabic body
- One consistent readable Arabic font
- Course title clearly distinct
- Module headings clearly distinct
- Reel headings visible but not oversized
- Project headings visually distinct from lessons
- Spoken paragraphs visually separated without excessive blank space
- No automatic numbering in spoken body
- No decorative punctuation
- No headers footers watermarks or tables unless explicitly requested

The document is a teleprompter source not a designed workbook

A module project may have a heading
The closure bridge inside it must not have its own heading

## Allowed document order

Use this order only

```text
COURSE TITLE
OPTIONAL SUBTITLE

MODULE 1 HEADING
REEL 1 HEADING
SPOKEN TEXT
REEL 2 HEADING
SPOKEN TEXT
PROJECT 1 HEADING
SPOKEN PROJECT INSTRUCTIONS
UNLABELLED MODULE CLOSURE BRIDGE

MODULE 2 HEADING
...

FINAL MODULE HEADING
...
FINAL RESOLUTION
```

Do not place the course map before the script
Do not place an appendix after the script
Do not place internal acceptance data in document properties visible to the user

## Final document verification

Perform all checks

1. Validate the DOCX archive and XML parts
2. Extract the visible text and compare it against the accepted course text
3. Verify module reel and project counts against the approved internal map
4. Verify every non final module has one project and one natural closure bridge
5. Verify the last project before the final module is the accepted compact integrated readiness proof
6. Verify the final module has complete resolution with no new promise critical standalone capability no project and no forward bridge after it
7. Verify no lesson project or closure line is missing truncated or duplicated
8. Verify every project ends at the learner action and upload instruction
9. Verify no rubric evaluator prompt scoring feedback routing attachment self evidence record or post upload logic leaked
10. Verify the strict evergreen firewall on the exact exported text
11. Verify no internal planning map research review or metadata leaked
12. Render every page
13. Inspect every rendered page visually
14. Rerun text gates after any formatting edit
15. Confirm the rendered text is the exact version that passed final meaning spoken variety module closure and evergreen gates

Do not deliver a DOCX that has not been rendered
Any language or text edit invalidates the prior render and requires extraction comparison rendering and visual inspection again

## Internal state boundary

The generation process may maintain thesis research map ledgers and findings while producing the course
These objects are temporary control state rather than user deliverables

Do not package them beside the DOCX
Do not create a handoff archive by default
Do not expose internal YAML JSON prompts reviewer reasoning source notes or project proof design

When the user later requests a revision reload any available internal state or rebuild the necessary minimum internally
The absence of an exported planning pack is not permission to weaken the revision

## Reference course use

When an approved course is supplied as a reference use it to calibrate

- Signal density
- Depth per lesson
- Natural lecturer stance
- Visual sentence organisation
- Project realism and proportional effort
- Progression from foundation to professional delivery
- Variety of openings closes module bridges and teaching forms

Do not infer universal defaults for

- Lesson count
- Module count
- Project length
- Tool mix
- Topic order
- Domain terminology
- Market examples
- Theory practice ratio
- Exact reel duration
- Module closure wording

The new course must still pass its own thesis capability market source project closure evergreen and delivery gates

## Domain transfer protocol

For every new course

1. Reset domain facts terminology examples market assumptions tool choices and all prior adapter assignments
2. Retain this standard and its quality gates
3. Build a new observable promise provisional performance episode map and capability decomposition internally
4. Assign provisional primary adapters only the necessary secondary adapters and active context overlays capability by capability
5. Research the new domain native performance practitioner reality language and authority requirements using the source firewall
6. Accept or replace every adapter assignment and derive any required custom adapter from verified practitioner performance
7. Build the capability graph capability mode matrix real performance or workflow trace coverage matrices duration budget project spine final boundary readiness record attachment self evidence ledger and module closure map internally
8. Ensure every promise critical capability is introduced and practised before the final boundary project and the final module is a resolution rather than a first proof gap
9. Validate representative lessons from different active adapters when the writer configuration or domain composition is unproven
10. Generate review rewrite domain neutrality render and verify the full course
11. Export one lecturer DOCX only

Do not preserve content from the previous domain in hidden context
Do not use design marketing sales language teaching or other prior domain examples in a new course merely because they existed in the reference
Do not carry a prior course adapter distribution into the new domain

## Version and provenance

Record internally

- Standard version
- Course version
- Generation date
- Market verification date
- Source versions when relevant
- Human overrides
- Quality gate results

Do not expose this metadata in the teleprompter
Do not use the generation date to justify volatile statements in the spoken course

## Non guarantee

This standard can enforce a strong evidence based production process
It cannot guarantee virality senior mastery employment revenue or performance without learner practice instructor delivery feedback market conditions and real execution

Never replace an honest course promise with an unprovable marketing guarantee
