# 07 TOOLS AI COST AND EVERGREEN DESIGN

## Principle before vendor

Teach the durable problem model decision and workflow before a specific tool implementation

For every tool lesson answer

- What category of problem does this tool solve
- What input quality does it require
- What output can and cannot be trusted
- What evidence controls its settings
- What failure modes expose misuse
- How can the learner verify the result
- What changes when a competing tool replaces it

Use one current tool as a worked example without making the course dependent on its brand unless the course promise is explicitly certification or mastery of that tool

## Tool specificity levels

### Stable primitive

Teach directly and deeply
Examples include layers functions variables campaign objectives or pronunciation contrasts when the primitive remains central across versions

### Current interface

Demonstrate only what is necessary
Teach how to locate the function when the interface changes

### Current specification

Verify from an official source near generation time
Teach the learner how to verify it again

### Vendor claim

Treat as untrusted until independently demonstrated
Do not repeat marketing language as capability truth

### Temporary trend

Include only when it changes current professional action and can be updated or removed without collapsing the course

## Evergreen split

Separate content internally into

- Core principles expected to remain useful
- Replaceable examples
- Current values requiring verification
- Optional market watch items

The main spoken course should be dominated by core principles and transferable workflows
Current values may appear when necessary but must carry an update trigger in `CLAIM_LEDGER`

## Cost aware professionalism

Design for the real learner budget without lowering the professional bar

For each tool or service classify

- Essential for learning
- Essential only for professional delivery
- Useful accelerator
- Optional convenience
- Replaceable example

Recommend payment when it materially improves one or more of

- Output fidelity
- Reliability
- Rights or commercial usage
- Collaboration
- Delivery speed
- Scale
- Data protection

Do not recommend a stack of paid subscriptions because each tool is fashionable
Do not describe all paid tools as unnecessary when real professional work depends on them

## AI role model

AI is a fallible collaborator and production component
It is not a magic replacement for domain judgement
It is not merely a novelty for personal content

Select roles appropriate to the course

- Clarifying an ambiguous brief
- Generating competing hypotheses
- Exposing missing questions
- Producing structured alternatives
- Drafting low risk first versions
- Generating or editing components
- Explaining unfamiliar material
- Reviewing against an explicit rubric
- Simulating practice or feedback
- Automating repetitive transformations
- Assisting debugging or diagnosis

Every AI use must preserve human responsibility for the consequential decision

## AI task contract

Before using a model define

```yaml
ai_task:
  purpose:
  supplied_context:
  constraints:
  desired_output:
  prohibited_inference:
  verification_method:
  privacy_class:
  rights_check:
  cost_limit:
  fallback:
```

Do not teach prompt wording without teaching the task specification behind it
Do not imply that one long prompt removes the need for evaluation

## AI verification

Require a verification method matched to the output

- Factual outputs against authoritative evidence
- Code through tests and runtime behaviour
- Visual outputs against fidelity anatomy text rights and brand constraints
- Language outputs against meaning register and natural usage
- Strategy outputs against assumptions evidence and competing hypotheses
- Operational outputs against policy data and human approval

Never use the same model response as both answer and sole proof

## AI limitations without corporate tone

Teach limitations through observable failure and professional consequence
Avoid abstract warnings such as use responsibly or avoid misinformation without a concrete action

Show

- What the failure looks like
- Why the model produced it
- What evidence catches it
- Which part may be safely reused
- When to switch to manual or specialist work

## Privacy rights and confidentiality

Do not place confidential client data personal data credentials unpublished work or restricted assets into an AI service without an approved data basis
Teach the learner to inspect service terms retention controls and commercial rights when the use affects professional delivery

Do not make legal certainty claims beyond authoritative evidence and the course scope
Use escalation rules for high consequence cases

## Tool independence test

For every tool dependent lesson ask

`If this tool disappears tomorrow what capability remains`

If the answer is none either

- Rewrite around the durable capability
- Mark the lesson as a replaceable current tutorial
- Remove it from the core

## Update design

Store current tool names interface paths prices specifications and policy notes separately from the core capability map when possible

An update should be able to replace a current example without rewriting the thesis project spine or causal explanations

Block export when an unstable value materially affects learner action and its current verification is missing
