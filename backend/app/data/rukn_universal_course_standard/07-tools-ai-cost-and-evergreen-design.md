# 07 TOOLS AI COST AND EVERGREEN SCRIPT DESIGN

## Principle before vendor

Teach the durable problem model decision and workflow before a specific tool implementation

For every tool lesson answer internally

- What category of problem does this tool solve
- What input quality does it require
- What output can and cannot be trusted
- What evidence controls its settings
- What failure modes expose misuse
- How can the learner verify the result
- What capability remains when the tool changes

Use a named tool when it is genuinely part of the promised work
Do not make the course a button tour when the real value is judgement workflow or diagnosis
Do not avoid a necessary action merely to sound timeless
Express the action through its stable purpose and observable result rather than through fragile interface coordinates

## Tool specificity levels

### Stable primitive

Teach directly and deeply
Examples include layers functions variables campaign objectives file states test conditions or pronunciation contrasts when the primitive remains central across versions

### Stable command or function

A tool name menu label or command may appear when it is established necessary and unlikely to change quickly
Explain what it does and how the learner knows it worked
Do not rely on memorising its screen position

### Fragile interface path

Do not place it in the recorded script
Replace click the third icon on the left or open the current menu path with the stable action goal visible state and a method for locating the function if the layout changes

### Current specification

Do not state the live value in the evergreen course when it can change quickly
Teach where and why the learner must verify the current specification at the moment of work
Use the current value only as internal research input

### Vendor claim

Treat as untrusted until independently demonstrated
Do not repeat marketing language as capability truth

### Temporary trend

Do not include it in the core recorded course
A trend that may disappear before the recording reaches the learner is not course knowledge
Teach the durable mechanism or decision behind it when that remains useful

## Strict evergreen script firewall

The final lecturer script must remain useful without routine rerecording caused by ordinary market tool or platform changes

The spoken course may contain

- Stable principles
- Transferable workflows
- Durable professional conventions
- Stable tool functions required by the promise
- Synthetic numerical examples clearly used to explain a method
- Variables the learner must inspect in their own live situation
- A durable method for checking a changing value when work requires it

The spoken course must not contain exact fast expiring details such as

- The current selling price of a service
- A recommended fixed client fee presented as a present market fact
- An application subscription price
- A plan name price quota discount or temporary free allowance
- A current platform commission fee or payment threshold
- A current salary market rate or exchange rate
- A current advertising cost benchmark
- A temporary algorithm rule reach trick ranking or platform preference
- A current follower count market statistic adoption percentage or trend number used as proof
- A current list of the best tools vendors or platforms
- A release specific feature claim likely to change
- A temporary file size duration usage or account limit
- A screen coordinate icon position or fragile menu path
- A statement whose correctness depends on words such as currently this month this year latest or at the moment
- A date bound legal policy or platform rule stated as a durable fact without teaching live verification

This prohibition applies even when the value is correct on generation day
A correct volatile fact still shortens the life of the recording
Apply the firewall to titles reel headings lecturer speech examples project instructions module closures and every other visible DOCX line

## Volatile fact transformation

When research finds a changing value do not copy it into the script
Transform it according to its teaching role

### Price or fee

Teach the variables method range construction decision boundary negotiation logic or verification source
Do not state the current amount

### Subscription or plan

Teach the capability the learner needs and how to compare the current available plans by required features limits rights and budget
Do not quote the current price or promise that a named tier contains a feature forever

### Platform limit or policy

Teach the operational consequence and the habit of checking the official live requirement before execution
Do not embed the current threshold in the recording

### Interface path

Teach the function purpose expected state and how to find the function by its meaning
Do not narrate a fragile sequence of screen positions

### Market statistic or ranking

Teach the mechanism decision evidence or comparison method the statistic was meant to support
Remove the ranking and temporary number

### Legal or regulated requirement

Teach the stable duty scope and verification or escalation method
Do not freeze a changeable clause threshold or procedure into an evergreen recording when the promise can be fulfilled without it
If the exact current rule is the course product itself this universal evergreen standard cannot honestly treat the product as permanent
Reframe the promise around interpretation verification and current application rather than memorising a frozen rule

## Evergreen claim state

Classify every material candidate statement before writing

```yaml
evergreen_claim:
  claim_id:
  teaching_role:
  stability: STABLE | SLOW_CHANGE | VOLATILE
  script_status: ALLOW | REWRITE_TO_METHOD | INTERNAL_ONLY | REMOVE
  expiry_trigger:
  durable_replacement:
  verification_action_if_needed:
  lesson_locations: []
```

Only `ALLOW` and an accepted `REWRITE_TO_METHOD` may enter the lecturer script
`INTERNAL_ONLY` may inform examples architecture or scope but may not be spoken
`REMOVE` must leave no trace in the course

Do not solve volatility by adding a recording date disclaimer
Do not say prices may change after quoting the price
Do not say check the latest after teaching a workflow that still depends on the old interface path
Rewrite the teaching so the volatile value is unnecessary

## Cost aware professionalism

Design for the real learner budget without lowering the professional bar

For each tool or service classify internally

- Essential for learning
- Essential only for professional delivery
- Useful accelerator
- Optional convenience
- Replaceable example

Recommend payment only at the principle level when it materially improves one or more of

- Output fidelity
- Reliability
- Rights or commercial usage
- Collaboration
- Delivery speed
- Scale
- Data protection

Do not quote the subscription amount
Do not recommend a stack of paid subscriptions because each tool is fashionable
Do not describe all paid tools as unnecessary when real professional work depends on them
Teach the learner how to judge whether the live offer is worth paying for in their own case

## AI role model

When AI is part of the course subject treat it as a fallible collaborator and production component
It is not a magic replacement for domain judgement
It is not merely a novelty for personal content

Select roles appropriate to the course

- Clarifying an ambiguous brief
- Generating competing hypotheses
- Exposing missing questions
- Producing structured alternatives
- Drafting low risk first versions
- Generating or editing components
- Explaining unfamiliar material
- Reviewing against a clear learner owned standard
- Simulating practice or feedback
- Automating repetitive transformations
- Assisting debugging or diagnosis

Every AI use must preserve human responsibility for the consequential decision
This section governs lessons in which the learner uses AI
It stops before any post upload platform behaviour

## AI task contract

Before teaching a model assisted task define internally

```yaml
ai_task:
  purpose:
  supplied_context:
  constraints:
  desired_output:
  prohibited_inference:
  verification_method:
  rights_or_confidentiality_boundary:
  cost_boundary:
  fallback:
```

Do not teach prompt wording without teaching the task specification behind it
Do not imply that one long prompt removes the need for domain verification
Do not export the internal task contract into the teleprompter
Teach only the learner action reasoning and verification needed for the promised skill

## AI verification

Require a verification method matched to the output

- Factual outputs against authoritative evidence
- Code through tests and runtime behaviour
- Visual outputs against fidelity anatomy text rights and brand constraints
- Language outputs against meaning register and natural usage
- Strategy outputs against assumptions evidence and competing hypotheses
- Operational outputs against policy data and human approval

Never use the same model response as both answer and sole proof
Teach limitations through observable failure and professional consequence rather than generic warnings

Show when relevant

- What the failure looks like
- Why the model may have produced it
- What evidence catches it
- Which part may be safely reused
- When to switch to manual or specialist work

## Professional boundaries in course content

Include rights confidentiality safety or escalation teaching only when it is part of performing the promised real skill
Explain the actual consequence and required action in ordinary practitioner language

Do not add platform policy text merely because the project will be uploaded
Do not add generic compliance paragraphs to every course
Do not turn the lecturer script into an application terms notice

## Tool independence test

For every tool dependent lesson ask

`If this tool changes tomorrow what capability remains`

If the answer is none either

- Rewrite around the durable capability
- Narrow the lesson to a stable function of the named tool
- Remove the lesson from the core

A tool specific course may teach the named product deeply
It must still prioritise stable functions states decisions and verification over temporary layout trivia

## Expiry resistance gate

Before export inspect every sentence that contains a number date currency plan tier fee version limit ranking current market claim or platform instruction

For each sentence ask

- Could this become wrong within a normal course lifetime
- Would a learner make a wrong decision if the value changed
- Can the same capability be taught through variables method conditions or live verification
- Does the sentence depend on a screen location rather than a stable function
- Is the value teaching the skill or merely making the course sound current

Rewrite or remove every sentence that fails

Block export when

- An exact service price subscription amount fee salary rate or temporary limit remains
- A current ranking trend statistic or platform preference carries the lesson argument
- A fragile interface path is required to follow the spoken text
- A disclaimer is being used instead of removing the volatile value
- The course would need routine rerecording only because ordinary prices plans interfaces or platform rules changed
- A current value is stored in a separate companion document intended to travel with the course

The goal is not to make the course vague
The goal is to make the teaching depend on durable judgement rather than disposable facts
